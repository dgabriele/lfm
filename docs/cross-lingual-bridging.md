# Cross-Lingual Bridging: System Design & Implementation

## Problem Statement

When an LLM learns a xenoglot (emergent IPA language from a frozen VAE decoder) alongside English, the two languages risk occupying **disjoint subspaces** in the model's representation geometry. The model can fluently generate in both languages but builds no internal pathway between them — the xenoglot's semantic structure and English's semantic structure never intersect. Translation requires that intersection.

Natural language solves this problem through discourse and elaboration: speakers bridge disparate conceptual domains via **analogical mechanisms** — structural or behavioral similarities that allow reasoning to transfer across contexts. We need architectural interventions that force the LLM to build analogous bridging representations.

## Architecture Overview

The solution has three components, designed to be applied as a **progressive curriculum**:

### Component 1: Cluster Anchors (Shared Pivot Tokens)

**File**: `src/lfm/translator/bilingual_corpus.py`
**Config**: `src/lfm/translator/config.py` → `BilingualCorpusConfig`

The embedding store partitions 300K sentence embeddings into **2048 semantic clusters** (stored in `cluster_labels.npy`). Each cluster covers ~146 embeddings from a semantically coherent region.

We inject cluster ID tokens `[C{id}]` into corpus documents. These tokens appear in **both** xenoglot and English contexts, forcing the model to build a representation for each `[C{id}]` that is useful in both language subspaces. The token becomes a **pivot point** — its embedding must lie in the geometric intersection of the two language subspaces, creating a bridge.

With 2048 anchors, the model has enough pivot points to tile the shared semantic space at meaningful granularity without being so fine-grained that each anchor becomes a mere lookup.

**Grounding**: Anchor-based alignment (Aldarmaki & Diab 2019), dictionary-based alignment analysis in mBERT (Pires et al. 2019).

### Component 2: Bilingual Documents (Shared Context Windows)

**File**: `src/lfm/translator/bilingual_corpus.py` → `BilingualCorpusGenerator`
**Config**: `src/lfm/translator/config.py` → `BilingualCorpusConfig`
**CLI**: `lfm translate generate-bilingual-corpus`

The existing `InterleavedDataset` mixes xenoglot and English at the **batch level** — each training example is entirely one language or the other. This prevents catastrophic forgetting but doesn't force cross-lingual attention within a single forward pass.

`BilingualCorpusGenerator` produces **single-line documents** where both languages share one context window:

```
[C1847] [T0] sə-kav-kos sɛj-sjes-tin-kaaɛ [T1] pa-ria-vse jɪɹ-zŋa-koem-ʕla [T2] ... [T3] ... [EN] The village market had been reorganized after the floods.
```

Under the **causal attention mask**, when the model predicts English tokens, it must attend to the cluster anchor and all xenoglot tokens. This forces cross-lingual representation alignment at the attention level — the model cannot predict the English continuation well unless it extracts semantic content from the xenoglot prefix.

#### Document Structure (left to right in the causal sequence):

1. `[C{id}]` — Cluster anchor. Shared across both language regions. Provides a "topic signal" the model can use as a routing cue.
2. `[T0] ... [T1] ... [T2] ... [T3] ...` — Four xenoglot dialogue turns with progressive elaboration. These are generated by running the source embedding through the trained `DialogueGame`'s generation pipeline (z-gen → Phase 1 decode → context transformer → next turn).
3. `[EN] ...` — The English source passage (from `passages.jsonl` in the embedding store). Truncated to `max_english_chars` (default 300) to keep the xenoglot/English balance within `max_len`.

#### Why Single-Line Format

The tokenizer (`TokenizedH5Dataset.from_corpus`) processes each non-blank line as a separate training example. Multi-line documents would be split across different training examples and shuffled apart. Single-line format guarantees both languages appear in the same attention context.

#### Data Flow

```
EmbeddingStore
  ├── embeddings.npy (N, dim) ──→ DialogueGame.generate() ──→ xenoglot turns
  ├── cluster_labels.npy (N,)  ──→ [C{id}] anchor token
  └── passages.jsonl           ──→ [EN] English passage text
                                        ↓
                              Single-line bilingual document
                                        ↓
                              bilingual_corpus_v7.txt
```

**Grounding**: Cross-lingual transfer correlates with shared context windows, not just shared training distribution (Conneau et al. 2020). Code-switching in multilingual models forces aligned representations at language boundaries.

### Component 3: Progressive Curriculum

**Configs**: `configs/pretrain_curriculum_phase{1,2,3}_v7.yaml`

The three phases are ordered by increasing cross-lingual demand. Each phase resumes from the previous phase's output model.

| Phase | Config | Data | What's Learned |
|-------|--------|------|----------------|
| 1 | `pretrain_curriculum_phase1_v7.yaml` | Pure xenoglot dialogue corpus | Phonotactic patterns, discourse structure, turn elaboration |
| 2 | `pretrain_curriculum_phase2_v7.yaml` | Batch-interleaved xenoglot (70%) + English (30%) | Both languages maintained, separate subspaces |
| 3 | `pretrain_curriculum_phase3_v7.yaml` | Bilingual documents with cluster anchors | Cross-lingual bridging representations |

#### Phase Transitions

- **Phase 1 → 2**: `model_name` in Phase 2 config points to `data/translator_curriculum_v7/phase1/model`. The model already knows xenoglot; now it relearns English without forgetting xenoglot (EMA + interleaving prevent catastrophic forgetting).
- **Phase 2 → 3**: `model_name` in Phase 3 config points to `data/translator_curriculum_v7/phase2/model`. The model knows both languages in separate subspaces; now bilingual documents force it to build bridging representations.

#### Hyperparameter Rationale

- **Phase 1**: `lr=5e-5`, no EMA. Standard pretraining on a new language. High gradient accumulation (`128`) compensates for `batch_size=1` (xenoglot sequences are long at `max_len=512`).
- **Phase 2**: `lr=1e-4`, EMA `decay=0.92`. Slightly higher LR because the model is adapting to mixed distribution. EMA prevents English knowledge from oscillating as the model encounters xenoglot batches.
- **Phase 3**: `lr=3e-5`, EMA `decay=0.95`. Lower LR because bridging should be a gentle refinement — the model already knows both languages, we're just teaching it to connect them. Higher EMA decay (more conservative averaging) preserves bridging representations once formed.

**Grounding**: Curriculum learning (Bengio et al. 2009), staged multilingual training.

## Implementation Details

### BilingualCorpusGenerator

**Inherits from**: `DialogueCorpusGenerator` (which inherits from `BaseCorpusGenerator`)

Reuses parent methods:
- `_load_dialogue_game()` — loads trained DialogueGame from checkpoint
- `_generate_documents()` — runs dialogue game's 4-turn generation pipeline
- `_load_embeddings()` — loads embedding store
- `_load_spm()` — loads sentencepiece model
- `decode_tokens_to_ipa()` — converts token tensors to formatted IPA strings

Adds:
- `_load_passages_and_clusters()` — loads `cluster_labels.npy` and `passages.jsonl` from embedding store
- `_format_bilingual_line()` — formats a single `[C{id}] [T0] ... [EN] ...` line
- `generate()` — overrides parent to produce single-line bilingual documents instead of multi-line dialogue documents

### BilingualCorpusConfig

```python
class BilingualCorpusConfig(LFMBaseConfig):
    dialogue_checkpoint: str = "data/dialogue_game/best.pt"
    decoder_path: str = "data/vae_decoder.pt"
    spm_path: str = "data/spm.model"
    embedding_store_dir: str = "data/embeddings"
    output_mode: str = "hyphenated_ipa"
    num_passes: int = 3          # Multiple passes with different seeds
    batch_size: int = 16
    max_english_chars: int = 300 # Truncate English to balance context window
    output_path: str = "data/translator/bilingual_corpus.txt"
    tokenize_model: str | None = None
    tokenize_max_len: int = 512
    device: str = "cuda"
    seed: int = 42
```

## Relationship to Existing Architecture

### What Existed Before

1. **`DialogueCorpusGenerator`** — generates multi-line xenoglot-only dialogue documents (4 turns, blank-line separated)
2. **`InterleavedDataset`** — batch-level mixing of xenoglot and English (each sample is one language)
3. **`SelfSupervisedTrainer`** — causal LM training with optional English interleaving and EMA
4. **`EmbeddingStore`** — 300K embeddings with 2048-cluster partition and passage text

### What Changed

- **No training loop modifications**. The bilingual corpus is just text — the existing `SelfSupervisedTrainer` and `TokenizedH5Dataset` handle it without changes.
- **No new dependencies**. BilingualCorpusGenerator reuses the dialogue game loading, generation, and IPA decoding from DialogueCorpusGenerator.
- **New data flow**: EmbeddingStore's cluster labels and passage text (previously only used by PairGenerator) are now used during corpus generation.

### What's Downstream

After Phase 3 pretraining, the model should have bridging representations. Evaluation approaches:

1. **Few-shot translation**: `<ipa> xenoglot </ipa> <eng>` prompting (existing eval pipeline in `evaluator.py`)
2. **Representation geometry**: Check whether xenoglot and English hidden states for same-embedding pairs are closer post-Phase-3 than post-Phase-2 (cosine similarity of final hidden states)
3. **Cluster anchor probing**: Check whether `[C{id}]` token representations are similar regardless of whether they appear in xenoglot or English context

## Future Extensions

### Chain-of-Analogy (deferred, builds on this foundation)

Once the model has bridging representations from Phase 3, a Phase 4 could train on explicit analogical reasoning chains:

```
[C1847] [T0] sə-kav-kos ... [Gloss] 3 phrases, falling sonority [Bridge] spatial-organization [EN] The village market...
```

The `[Gloss]` layer (phonotactic description) and `[Bridge]` layer (semantic keywords) are computable from existing data — no human annotation needed. This would give the model a **learned procedure** for traversing the bridge, not just the bridge itself.

### Contrastive Representation Alignment (alternative to implicit bridging)

If Phase 3's implicit bridging via shared attention is insufficient, an auxiliary contrastive loss could directly pull together hidden representations of paired xenoglot/English sequences:

```python
# Pull together representations of semantically paired sequences
z_x = normalize(xenoglot_hidden.mean(dim=1))
z_e = normalize(english_hidden.mean(dim=1))
loss_align = cross_entropy(z_x @ z_e.T / temperature, arange(B))
```

This would require modifying `SelfSupervisedTrainer._train_epoch()` to occasionally form paired batches.

### Inference-Time Thought Loops

At inference, the sentence-transformer that produced the original embeddings provides a **computable distance metric** for self-verification:

1. Generate English interpretation from xenoglot input
2. Re-encode interpretation with sentence-transformer
3. Measure cosine distance to original embedding
4. If distance exceeds threshold, feed back and refine

This requires no training changes — only an inference-time generation pipeline.
